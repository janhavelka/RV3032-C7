# RV3032-C7 2.0.0 implementation report - 2026-07-13

## Scope and evidence policy

This report was created before implementation-source edits and was completed
after the implementation and verification pass. It records both the audited
baseline and the final evidence. Three delegated audits covered lifecycle/jobs,
persistence/integration, and vendor capability semantics; their findings and
focused changes were checked against the source and integrated by the primary
agent.

Hardware execution, device flashing, physical EEPROM commands, publishing,
commits, tags, and edits to the sibling TunnelMonitor repository were outside
this task. Hardware-in-the-loop status is **NOT RUN**.

- Repository HEAD at start: `8b9e7fb`; declared version: `1.6.0`.
- Native baseline: `python -m platformio test -e native` - 56/56 passed.
- ESP32-S3 baseline: passed; RAM 22,368 bytes, flash 421,086 bytes.

## Vendor interpretation record

The local RV-3032-C7 application manual Rev. 1.3 and datasheet were inspected,
including the prompt's required register, EEPROM, timing, status, password,
PMU, and primary-cell pages.

- Primary-cell target: `(persistentC0 & 0x4C) | 0x20`.
- `TCM=00` disables the charger. `TCR` is preserved because its value is
  electrically inactive while TCM disables charging.
- On application-manual page 141, `TCR=00` is printed in black as the unchanged
  delivery value, while `TCM=00` is red as the required primary-cell policy.
  The driver generalizes that example by preserving an existing inactive TCR
  selection rather than resetting it.
- Status is write-zero-to-clear for PORF/VLF/UF/TF/AF/EVF, while every Status
  write also clears THF/TLF. The fake and typed clear paths model that side
  effect explicitly.
- EECMD is write-only/read-zero. Active C0-CA mirrors and persistent C0-EA
  EEPROM are independent storage.
- Direct persistence uses `READ_ONE=0x22` and `WRITE_ONE=0x21`; generic
  persistence and primary ensure do not issue `UPDATE_ALL=0x11` or
  `REFRESH_ALL=0x12`.
- Both documented disabled BSM encodings are accepted by the native fake, but
  every driver EECMD path first proves the required BSM00/TCM00 access state.

## API and migration outcome

Version 2.0.0 intentionally removes `Config::backupMode` and the unsafe
`setPrimaryBatteryBackupDefaults()` helper. It adds the yielding `waitMs`
callback, append-only error codes, typed cooperative results, persistent
inspection/user EEPROM/password APIs, and the explicit
`ensurePrimaryCellConfiguration()` operation. Existing feature names remain
where their hardware capability remains safe, but multi-transfer setters now
return IN_PROGRESS on admission and require `pollJob()`; opt-in persistence is
then advanced through `tick()` or `pollEeprom()`.

## TunnelMonitor compatibility audit

No TunnelMonitor file or dependency pin was changed. The library boundary now
supports the planned integration profile: generic persistence can remain
disabled, ordinary work can be serialized by the application's I2C owner with
`pollJob(now, 1, used)`, and the owner may explicitly invoke the bounded primary
ensure once during startup. A TunnelMonitor build and hardware integration were
not run in this repository.

## Intentional omissions

- No physical HIL, upload, EEPROM endurance, voltage/backfeed, power-cycle, or
  retention claim.
- No library-owned bus recovery or replay of a possibly mutating callback.
- No general synchronous multi-callback API beyond primary-cell ensure.
- No update-all/refresh-all use in persistence or primary provisioning.
- No password guessing API, password readback, or unrestricted EECMD surface.
- No commit, tag, release publication, or dependency-pin update.
